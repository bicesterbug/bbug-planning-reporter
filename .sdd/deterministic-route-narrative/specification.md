# Specification: Deterministic Route Narrative

**Version:** 1.0
**Date:** 2026-02-19
**Status:** Implemented

---

## Problem Statement

The `route_narrative` field in the review output is always null because it depends on the LLM populating an optional `route_assessment` field during the structure call. The LLM consistently skips this field because the JSON tool schema marks it as optional with a null default — the schema semantics override the prompt's "MUST populate" instruction. The route assessment data needed for the narrative already exists as structured data in `self._route_assessments`, making LLM generation unnecessary.

## Beneficiaries

**Primary:**
- Website consumers — `route_narrative` provides per-destination route summaries (distance, score, rating) in the review JSON, allowing the website to display route quality without fetching the separate routes JSON file

**Secondary:**
- Token budget — removing the `route_assessment` field from the structure prompt and schema reduces the LLM's tool schema size and prompt length, saving input tokens on every review

---

## Outcomes

**Must Haves**
- `route_narrative` is reliably populated in every review that has route assessment data
- The data is built deterministically from the existing structured route assessment results, not generated by the LLM
- The `route_assessment` field is removed from the structure call schema and prompt (no longer needed)

**Nice-to-haves**
- None

---

## Explicitly Out of Scope

- Changing the `_routes.json` file format or content
- Changing the report prompt's Route Assessment section (the markdown report still renders route data from evidence context)
- Adding new fields to the route_narrative data shape beyond what already exists
- Generating prose narratives — the narrative field is removed, replaced by structured data only

---

## Functional Requirements

**FR-001: Build route_narrative deterministically from route assessment data**
- Description: The orchestrator must build the `route_narrative` dict directly from `self._route_assessments` after route assessment completes, without any LLM involvement. For each destination, the narrative includes the destination name, shortest and safest route summaries (distance_m, ltn_score, rating), and the same_route flag.
- Acceptance criteria: When a review completes with route assessments, the `route_narrative` field in the review JSON is a dict with a `destinations` array. Each entry has `destination_name`, `shortest_route_summary` (with `distance_m`, `ltn_score`, `rating`), `safest_route_summary` (same fields), and `same_route`. The `narrative` string field is removed from each destination entry.
- Failure/edge cases: When no route assessments exist (`self._route_assessments` is empty), `route_narrative` is None. When a destination has `same_route: true`, both summaries contain the same values.

**FR-002: Remove route_assessment from ReviewStructure schema**
- Description: The `route_assessment` field, `RouteAssessmentSection`, `RouteDestinationItem`, and `RouteDestinationSummary` models must be removed from the ReviewStructure Pydantic model in `review_schema.py`. The structure call no longer asks the LLM to produce route data.
- Acceptance criteria: `ReviewStructure` has no `route_assessment` attribute. The JSON tool schema sent to the LLM does not include a `route_assessment` property. The `RouteAssessmentSection`, `RouteDestinationItem`, and `RouteDestinationSummary` classes no longer exist in `review_schema.py`.
- Failure/edge cases: Old structure call responses that happen to include `route_assessment` must not cause validation errors — Pydantic ignores extra fields by default.

**FR-003: Remove route_assessment guidance from structure prompt**
- Description: The `route_assessment` field guidance block must be removed from the structure prompt in `structure_prompt.py`. This eliminates the tokens spent describing the field and the contradictory "MUST populate" instruction.
- Acceptance criteria: The structure prompt system message does not contain "route_assessment". The user prompt is unchanged.
- Failure/edge cases: None.

**FR-004: Remove narrative field from route_narrative data shape**
- Description: The per-destination `narrative` string field (previously LLM-generated prose) must be removed from the `route_narrative` data shape. The remaining fields (`destination_name`, `shortest_route_summary`, `safest_route_summary`, `same_route`) are sufficient for the website.
- Acceptance criteria: The `route_narrative.destinations[].narrative` field does not exist in the review JSON output. The API schema reflects the removal.
- Failure/edge cases: Old reviews stored in Redis with a `narrative` field in route_narrative should still deserialise without error.

## QA Plan

**QA-01: Verify route_narrative populated in review JSON**
- Goal: Confirm the review JSON contains route_narrative with per-destination data
- Steps:
  1. Submit a review for an application with route destinations configured
  2. Wait for completion
  3. Download the `_review.json` from the output URL
  4. Inspect `review.route_narrative`
- Expected: `route_narrative` has a `destinations` array with entries for each assessed destination. Each entry has `destination_name`, `shortest_route_summary` (with `distance_m`, `ltn_score`, `rating`), `safest_route_summary` (same fields), and `same_route`. No `narrative` string field.

**QA-02: Verify route_narrative absent when no routes**
- Goal: Confirm reviews without route assessment have null route_narrative
- Steps:
  1. Submit a review for an application with no route destinations
  2. Wait for completion
  3. Download the `_review.json`
- Expected: `route_narrative` is null.

**QA-03: Verify structure call no longer includes route_assessment**
- Goal: Confirm the LLM structure call schema is smaller
- Steps:
  1. Check worker logs for the structure call
  2. Inspect the tool schema sent to the LLM
- Expected: No `route_assessment` property in the tool schema. No `RouteAssessmentSection` or `RouteDestinationItem` in `$defs`.

## Appendix

### Glossary
- **route_narrative**: A structured summary of route assessments included in the review JSON, with per-destination distance, score, and rating
- **route_assessment**: The (now removed) optional field in the structure call schema that asked the LLM to produce route data — replaced by deterministic build
- **structure call**: The first LLM call that produces structured JSON data (ratings, aspects, compliance) via tool_use
