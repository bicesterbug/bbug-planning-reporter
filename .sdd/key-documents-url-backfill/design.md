# Design Document

## Architecture Overview

### Current Architecture Context
- The review pipeline ingests planning documents then calls Claude with a free-text listing. Claude produces a `ReviewSchema` whose `key_documents` entries carry LLM-echoed `title` and `url`; these flow straight through the orchestrator into the `GET /api/v1/reviews/{id}` response as `KeyDocument` objects.
- Because `url` is echoed from free-text prose, it is sometimes `null` or wrong. A stable 12-char MD5 `document_id` is already generated by the scraper (`parsers.py:638-641`) but not currently surfaced to Claude or the backfill site.
- Requirements below are derived from the research document's "Proposed Approach" and "Integration Points" sections — there is no formal specification. FR numbering (FR-001..FR-007) originates in this design document; it is not imported from any upstream spec.

### Proposed Architecture
- Persist `document_id` in `IngestionResult.document_metadata` entries (existing `local_path` keying retained; `document_id` added as a value field).
- Show `document_id` to Claude in the ingested documents prompt text; drop `url` from that text.
- Require Claude to echo `document_id` on each `KeyDocumentItem` instead of composing `title`/`url`.
- The backfill helper builds the `{document_id: metadata}` index internally on each call and deterministically backfills `title` from `description` and `url` from the stored URL before producing `KeyDocument` response objects.
- On mismatch: keep the entry, set `title="(unknown document)"` and `url=null`, emit `logger.warning(...)`. Claude is never a title source.

### Technology Decisions
- `document_metadata` keeps `local_path` keying to minimise blast radius on existing readers; the `{document_id: metadata}` index is derived ad-hoc inside the backfill helper on each call.
- Deterministic post-processing layered on structure output matches existing codebase convention (e.g. `coerce_category`).

### Quality Attributes
- **Maintainability:** lookup transformation localised in one helper; schema and prompt evolve together.
- **Reliability:** public `KeyDocument` contract unchanged; `url` becomes reliably populated for every ingested document.

## API Design

**Public contract (unchanged):** `KeyDocument` in `src/api/schemas.py:166-188` continues to expose `title: str` and `url: str | None`. The wire shape returned from `GET /api/v1/reviews/{review_id}` does not change.

**Internal contract (changed):** `KeyDocumentItem` in `src/agent/review_schema.py:77-98` gains a required `document_id: str` field. `title` and `url` are removed from `KeyDocumentItem` entirely — Claude no longer produces them. The authoritative source of truth for `title` and `url` becomes `IngestionResult.document_metadata`.

**Error strategy:** A `document_id` echoed by Claude that does not match any ingested document is logged via `logger.warning(...)` but does NOT drop the entry — the entry is emitted with `url=None` and a fallback `title`.

## Added Components

### Key-document backfill helper
**Description** Given the `KeyDocumentItem` list from the structure call and the `document_metadata` dict, builds a `{document_id: metadata}` index internally and returns `KeyDocument` objects with `title`/`url` sourced from metadata. On mismatch: keeps the entry with `url=None` and `title="(unknown document)"` (Claude is never a title source), emits `logger.warning(...)`. If the `{document_id: metadata}` index dict-comp encounters duplicates, the later entry wins — this is expected behaviour; MD5 collisions and duplicate source URLs are not expected in practice.

**Users** Orchestrator, at the existing passthrough site (orchestrator.py:1838-1846).

**Kind** Function (private helper)

**Location** `src/agent/orchestrator.py` (adjacent to existing review-assembly helpers)

**Details**
```
def _backfill_key_documents(
    items: list[KeyDocumentItem],
    document_metadata: dict[str, dict],
    review_id: str,
    application_ref: str,
) -> list[KeyDocument]:
    index = {m["document_id"]: m for m in document_metadata.values()}
    out = []
    for item in items:
        meta = index.get(item.document_id)
        if meta is None:
            logger.warning(
                "key_document id not in metadata",
                review_id=review_id,
                application_ref=application_ref,
                document_id=item.document_id,
            )
            out.append(KeyDocument(title="(unknown document)", url=None,
                                   category=item.category, summary=item.summary))
            continue
        out.append(KeyDocument(title=meta["description"], url=meta["url"],
                               category=item.category, summary=item.summary))
    return out
```

**Requirements References**
- `key-documents-url-backfill:FR-005`: deterministic backfill of `title`/`url` from metadata via `document_id`.
- `key-documents-url-backfill:FR-006`: mismatch emits warning and produces `url=None` entry.
- `key-documents-url-backfill:FR-007`: public `KeyDocument` shape is preserved.

**Test Scenarios**

**TS-06: successful backfill**
- Given: `KeyDocumentItem(document_id="abc", category="policy", summary="...")` and metadata containing `abc` → `{"description": "NPPF Dec 2024", "url": "https://...", ...}`
- When: `_backfill_key_documents([item], metadata)` is called
- Then: result has one `KeyDocument` with `title="NPPF Dec 2024"` and `url="https://..."`

**TS-07: mismatch fallback**
- Given: `KeyDocumentItem(document_id="xxx")` and metadata that does not contain `xxx`
- When: `_backfill_key_documents` is called
- Then: output contains one entry with `url is None`, a warning is logged, and the entry is NOT dropped

**TS-08: multiple entries with one mismatch**
- Given: three items, two with matching ids and one without
- When: helper runs
- Then: output has three entries; the two matches have populated `url`, the mismatch has `url is None`

## Test Feasibility
- All new behaviour is exercisable with unit tests around `_backfill_key_documents` plus existing orchestrator integration tests updated to populate `document_id` on fixtures. No QA plan is attached (no spec); unit + existing integration coverage are sufficient.
- Existing test `tests/test_orchestrator.py:1049` already asserts `url is not None` — it will strengthen under this change.
- Tests that directly construct `KeyDocumentItem` need a `document_id` supplied.

## Risks and Dependencies
- **LLM hallucinating a `document_id`:** the id is an opaque 12-char hash, so hallucination of a valid-looking-but-wrong id is unlikely. Mitigated by the warning-log + `url=None` fallback (FR-006).
- **Prompt change may affect doc selection quality:** removing the URL column and adding an id prefix is a small change; summary/category behaviour is expected to be unchanged. Monitor review quality on first run. Rollback is a one-line prompt-string revert.
- **Test updates:** callers that construct `KeyDocumentItem` directly must add `document_id`. This is a mechanical fix.
- **Dependency:** requires `document_id` to be present on every selected document at ingestion time — it already is (scraper generates it).
- **Missing `document_id` on a selected document:** should not happen given FR-001, but if an upstream bug results in a selected document reaching the metadata population sites without a `document_id`, population code would raise a `KeyError`. Mitigation: use `doc.get("document_id", "")` at the population sites and let the backfill mismatch path handle the resulting empty id (warning + `url=None`).

## Appendix

### Glossary
- **`document_id`:** 12-char MD5 hash of a document's source URL, generated by the Cherwell scraper; stable across runs for the same source URL.
- **Backfill:** deterministic post-processing step that populates fields on structured-output objects using an authoritative source, replacing LLM-echoed values.
